## 📌자율적인 책임

- 객체지향 공동체를 구성하는 기본 단위는 ‘자율적’인 객체임
- 자율적인 객체란 스스로 정한 원칙에 따라 판단하고 스스로의 의지를 기반으로 행동하는 객체임.
- ‘책임’이란 요청을 처리하기 위해 객체가 수행하는 행동임
- 객체가 책임을 자율적으로 수행하기 위해서는 객체에게 할당되는 책임이 자율적이어야 함.
- 너무 상세한 수준의 책임들은 객체의 자율성을 제한함. 또한 협력의 의도를 명확하게 표현하지 못할 정도로 추상적인 것도 문제임.
    
    → 책임은 협력에 참여하는 의도를 명확하게 설명할 수 있는 수준 안에서 추상적이어야 함.
    
- 특징
    - 객체가 ‘어떻게’ 해야 하는가가 아니라 ‘**무엇’을 해야 하는가**를 설명하는 것.
- 객체가 자신에게 할당된 책임을 수행하도록 만드는 것은 외부에서 전달되는 요청(메시지)임.

## 📌메세지와 메서드

### 메세지

- 메세지는 객체가 다른 객체에 접근하는 방법임.
- 메세지-전송은 객체의 행동을 유발하는 행위임.
- 메세지에는 메세지 이름이 있고, 예를 들면 "왕이 모자 장수에게 증언하라"에서 "증언하라"가 메세지 이름임.
- 메세지를 전송할 때 추가 정보가 필요한 경우 메세지의 인자(argument)를 통해 제공할 수 있음.
- 객체가 수신할 수 있는 메세지의 모양이 객체가 수행할 책임의 모양을 결정함.

⇒ 객체가 유일하게 이해할 수 있는 의사소통 수단은 메세지뿐이며 객체는 메세지를 처리하기 위한 방법을 자율적으로 선택할 수 있음.

⇒ 외부의 객체는 메세지만 볼 수 있고 객체 내부를 볼 수 없기 때문에 객체의 외부와 내부가 분리됨.

### 메서드

- 메세지를 처리하기 위해 내부적으로 선택하는 방법
- 메세지를 처리할 수 있다고 판단되면 자신의 책임을 수행하기 위해 메세지를 처리할 방법인 메서드를 선택함
- 메세지를 수신한 객체가 실행 시간에 메서드를 선택할 수 있다는 것은 다른 프로그래밍 언어와 객체지향 프로그래밍 언어를 구분 짓는 핵심적인 특징 중 하나임.
    
    ⇒ 프로시처 호출에 대한 실행 코드를 컴파일 시간에 결정하는 절차적인 언어와 확연히 구분되는 특징임.
    

### 다형성

- 서로 다른 유형의 객체가 동일한 메세지에 대해 서로 다르게 반응하는 것
    
    ⇒ 서로 다른 타입에 속하는 객체들이 동일한 메세지를 수신할 경우 서로 다른 메서드를 이용해 메세지를 처리할 수 있는 메커니즘
    
- 하나의 메세지와 하나 이상의 메서드 사이의 관계
- 역할, 책임, 협력과 깊은 관련
    - 메세지 송신자의 관점이 중요함
    - 송신자의 관점에서 다형적인 수신자들을 구별할 필요가 없으며 자신의 요청을 수행할 책임을 지닌다는 점에서 모두 동일함.
    - 동일한 역할을 수행하는 다양한 타입의 객체와 협력할 수 있게 함
- 동일한 역할을 수행할 수 있는 객체들 사이의 **대체가능성**을 의미함.
    
    → 설계를 유연하고 재사용 가능하게 만듦
    
- 수신자의 종류를 캡슐화함
    - 송신자와 수신자 간의 객체 타입에 대한 결합도를 메세지에 대한 결합도를 낮춤으로써 달성됨

### 유연하고 확장 가능하고 재사용성이 높은 협력의 의미

1. 협력이 유연해짐
    
    → 송신자는 수신자에 대해 어떤 가정도 하지 않음, 메세지를 이해만 하면 됨
    
    → 송신자에 대한 파급효과 없이 유연하게 협력 변경 가능
    
2. 협력이 수행되는 방식 확장 가능
    
    → 송신자에게 아무런 영향을 미치지 않고 수신자 교체 가능하기 때문에 협력의 세부적인 수행 방식을 쉽게 수정 가능
    
3. 협력이 수행되는 방식 재사용 가능
    
    → 수신자의 대체 가능성 때문
    

⇒ **메세지가** 수신자와 송신자 사이의 결합도를 낮춤으로써 설계를 유연하고, 확장 가능하고 재사용 가능하게 만듦. 

## 📌메세지를 따라라

- 객체지향 애플리케이션의 중심 사상은 연쇄적으로 메세지를 전송하고 수신하는 객체들 사이의 협력 관계를 기반으로 사용자에게 유용한 기능을 제공하는 것임.
- 클래스를 정의하는 것이 먼저가 아니라 객체들의 속성과 행위를 식별하는 것이 먼저임. 클래스는 객체의 속성과 행위를 담는 틀일 뿐임.
- 클래스를 중심에 두는 설계는 유연하지 못하고 확정하기 어려움
- 객체가 메세지를 선택하는 것이 아닌 메세지가 객체를 선택하게 해야 함. 이를 위해서는 메세지를 중심으로 협력을 설계해야 함.
- 책임 주도 설계
    - 책임을 완수하기 위해 협력하는 객체들을 이용해 시스템을 설계하는 방법
    - 객체들 간에 주고받는 메세지를 기반으로 적절한 역할, 책임 협력을 발견하는 것
    - 메세지가 수신자의 책임을 결정함
    - What/Who 사이클 : 어떤 행위가 필요한가를 결정한 후에 이 행위를 수행할 객체를 결정하는 것
    - 협력이라는 문맥 안에서 필요한 메세지를 먼저 결정한 후 메세지를 수신하기에 적합한 객체를 선택함. 그리고 수신된 메세지가 객체의 책임 결정.
    - 메세지가 결정된 후에야 메세지를 수신할 후보를 선택하는 것으로 초점 이동
- 묻지 말고 시켜라
    - 메세지를 먼저 결정하고 객체가 메세지를 따르게 하는 설계방식 → 객체가 외부에 제공하는 인터페이스가 독특한 스타일을 따르게 함.
        
        ⇒ **(Tell, Don’t Ask) or 데메테르 법칙**
        
    - 객체는 다른 객체의 결정에 간섭하지 말아야하며, 모든 객체는 자신의 상태를 기반으로 스스로 결정을 내림
- 메세지를 믿으면 자율적인 책임은 저절로 따라올 것임.

## 📌객체 인터페이스

- 인터페이스는 두 사물이 상호작용 할 수 있게 이어주는 방법이나 장치를 의미
- 인터페이스의 3가지 특징
    1. 인터페이스의 사용법을 익히기만 하면 내부 구조나 동작 방식을 몰라도 쉽게 대상을 조작하거나 의사를 전할 수 있음
    2. 인터페이스 자체는 변경하지 않고 단순히 내부 구성이나 작동 방식만 변경하는 것은 인터페이스 사용자에게 어떤 영향도 미치지 않음
    3. 대상이 변경되더라도 동일한 인터페이스를 제공하기만 하면 아무런 문제 없이 상호작용 할 수 있음
- 객체의 인터페이스는 객체가 수신할 수 있는 메시지의 목록으로 구성되며, 객체 간 상호작용은 메시지 전송을 통해 이루어짐.
- 인터페이스는 내부와 공용으로 구분되며, **공용 인터페이스**는 내부에서만 접근 가능한 사적인 인터페이스와 구분하기 위해 외부에 공개된 인터페이스를 의미함.
- 객체가 수신 가능한 메시지가 인터페이스를 결정하며, 이는 객체가 어떤 책임을 수행할 수 있고 어떤 메시지를 처리할지를 나타냄.
- 메시지로 구성된 공용 인터페이스는 객체의 외부와 내부를 명확하게 분리하여 객체지향의 강점을 제공함.

## 📌인터페이스와 구현의 분리

- 훌륭한 객체란 구현을 모른 채 인터페이스만 알면 쉽게 상호작용할 수 있는 객체를 의미
    
    → 이것은 객체를 설계할 때 공용 인터페이스와 구현을 명확하게 분리해서 고려해야 한다는 것을 의미
    
    ⇒ 이를 인터페이스와 구현의 분리 원칙이라고 함.
    
- 왜 중요한가
    - 소프트웨어는 항상 변경되기 때문
    - 변경을 관리하기 위한 것
- 이 원칙을 수행하기 위한 객체 설계  방법을 **캡슐화**라고 함.

### 캡슐화

- 객체의 자율성을 보존하기 위해 구현을 외부로부터 감추는 것
- 두가지 관점에서 사용
    - 상태와 행위의 캡슐화
    - 사적인 비밀의 캡슐

## 📌책임의 자율성이 협력의 품질 결정

1. 자율적인 책임은 협력을 단순하게 만든다.
2. 자율적인 책임은 모자 장수의 외부와 내부를 명확하게 분리한다.
3. 책임이 자율적일 경우 책임을 수행하는 내부적인 방법을 변경하더라도 외부에 영향을 미치지 않는다.
4. 자율적인 책임은 협력의 대상을 다양하게 선택할 수 있는 유연성을 제공한다.
5. 객체가 수행하는 책임들이 자율적일수록 객체의 역할을 이해하기 쉽다.
